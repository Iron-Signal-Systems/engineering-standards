# Release Publication

## Purpose

Release publication makes one already accepted, signed, and deterministically
produced ISRAS release available through GitHub without changing its source or
artifact identity.

Publication is a separate authority boundary. It does not create or push a Git
tag, move `main`, modify a consuming project, initialize a project, or rebuild an
artifact. Those operations require their own reviewed stages.

## Command

Build the publication command from committed source:

```bash
./tools/build-release-publication.sh
```

Perform the read-only publication preflight:

```bash
./.local/bin/isras-release-publication check \
  --version 0.1.1
```

After reviewing the exact source, signed remote tag, local artifacts, build
evidence, notes, and retained preflight evidence, explicitly publish:

```bash
./.local/bin/isras-release-publication publish \
  --version 0.1.1 \
  --confirm
```

`publish` is the only command in this implementation with GitHub Release
creation and asset-upload authority. The legacy `isras-release publish` entry
point is disabled and cannot bypass this boundary.

## Required source boundary

Both `check` and `publish` require:

- the canonical Engineering Standards repository;
- the configured authoritative branch, normally `dev`;
- a completely clean repository, including no untracked files;
- an exact nonzero 40-character source commit;
- a valid signature on that exact commit;
- local `HEAD` equal to the authoritative remote branch tip;
- a stable `MAJOR.MINOR.PATCH` value in `VERSION`;
- a matching immutable `isras-vMAJOR.MINOR.PATCH` identity;
- matching committed changelog and release-note content;
- a credential-free canonical GitHub origin; and
- no preexisting draft or published GitHub Release for the selected tag,
  established through a complete paginated release inventory.

A release name is never reused. An existing draft or published release is a hard
failure and is not edited, replaced, or deleted by a new invocation.

## Remote tag boundary

Publication never creates or pushes a tag. Before any release mutation, the
command reads the remote Git tag through GitHub and requires:

- an annotated tag object rather than a lightweight tag;
- the expected tag name;
- a direct commit target equal to the exact source commit;
- GitHub signature verification status `verified=true`;
- verification reason `valid`; and
- nonempty signature, payload, and verification-time evidence.

The same source and remote-tag boundary is checked again after publication.

## Local artifact boundary

The publication input is the deterministic six-file output from
[`RELEASE-ARTIFACT-PRODUCTION.md`](RELEASE-ARTIFACT-PRODUCTION.md):

```text
isras-validator-linux-amd64
isras-project-framework.tar.gz
isras-contracts.tar.gz
provenance.json
SHA256SUMS
SHA512SUMS
```

Publication requires the private `artifact-build.json` evidence generated by the
producer. It independently requires:

- exact version, tag, source repository, and source commit identity;
- exactly six sorted artifact records;
- exact names, kinds, platform identity, sizes, SHA-256 values, and SHA-512
  values;
- an exact regular-file directory with no extra entry or symbolic link;
- both checksum manifests matching the build evidence and observed bytes;
- provenance matching the release and core artifact bytes; and
- the validator binary's linker-bound identity matching the release source.

The source, notes, build evidence, and artifact directory must remain inside the
canonical repository boundary. The complete local boundary is checked before
remote work, immediately before publishing the verified draft, and after final
publication.

## Draft-first publication

Publication uses a draft-first transaction:

1. create one empty draft for the exact existing tag, title, notes, and source
   commit;
2. verify that the new draft is empty, non-prerelease, and unpublished;
3. upload each of the six declared assets without replacement or clobbering;
4. verify every upload response against expected name, state, size, and GitHub
   SHA-256 metadata;
5. re-read the complete draft and reject missing, extra, duplicate, unavailable,
   or altered assets;
6. download every remote asset into a private bounded temporary directory;
7. verify downloaded SHA-256, SHA-512, manifests, provenance, and exact inventory;
8. revalidate the complete local source and artifact boundary;
9. change only the verified draft state to published; and
10. re-read, re-download, and reverify the published release and remote tag.

The command never uses an asset-clobber option. It never accepts a partial asset
set as a release.

## Bounded transport

GitHub API operations use the authenticated `gh` command. Publication does not
read, display, or retain the authentication token itself.

Captured command output is bounded and credential-shaped diagnostics are
censored before they enter errors or evidence. Remote asset downloads are
bounded to the maximum accepted release-asset size before they are hashed.

Release-asset byte uploads use the authenticated GitHub CLI
`gh release upload` command without `--clobber`. The CLI resolves the selected
draft and its GitHub-provided upload URL, which uses the `uploads.github.com`
release-upload service. Publication re-reads the exact draft by release ID after
every upload and verifies the expected name, state, size, and SHA-256 digest.
Tests must reject the defective `gh api --hostname uploads.github.com` invocation,
which derives the invalid host `api.uploads.github.com`.

## Failure and cleanup

After the command creates a draft, any failure before publication triggers
cleanup only when GitHub still proves that the release is:

- the exact draft created by the current invocation;
- bound to the expected tag and title;
- still unpublished and non-prerelease; and
- empty or containing only an unaltered subset of the six declared assets.

Only that exact draft may be deleted automatically. Cleanup reads the draft by
its exact release ID, verifies the complete identity and allowed asset subset,
deletes that ID, and requires an authenticated ID-based absence check afterward.
A tag-based public release lookup is not cleanup evidence because it may not
expose drafts. The Git tag is never deleted.
A published release, a differently identified draft, or a draft containing an
unknown or altered asset is preserved for investigation rather than removed.

If a publication request reports an error but the authoritative GitHub state
proves the exact release was published, the command continues to final
verification rather than blindly retrying the write.

## Evidence

Private JSON and text evidence is written atomically under:

```text
.local/validation/releases/isras-vMAJOR.MINOR.PATCH/publication/
```

JSON evidence conforms to
`schemas/isras-release-publication-v1.schema.json` and records:

- source, version, tag, repository, notes, artifact, and build-evidence identity;
- every publication stage decision;
- release and remote-asset identifiers;
- complete expected SHA-256 and SHA-512 values;
- remote sizes and SHA-256 metadata;
- upload and remote-download verification status;
- cleanup status;
- final release URL when publication succeeds; and
- a censored bounded failure reason when it does not.

Evidence files use mode `0600`; containing directories use mode `0700`.
Evidence-write failure invalidates the operation. When a draft exists, a failure
to retain evidence also enters the same safe-cleanup decision boundary.

## Acceptance boundary

This implementation is accepted only when tests prove:

- read-only `check` performs no remote write;
- only the separately named `publish --confirm` operation has release authority;
- existing drafts and published releases are never overwritten;
- lightweight, unsigned, invalid, or commit-drifted remote tags fail closed;
- build-evidence, manifest, provenance, validator, source, title, and notes drift
  fail closed;
- upload inventory is exact and no asset is clobbered;
- remote metadata and downloaded bytes are both verified before publication;
- altered remote bytes fail and an exact incomplete draft is removed safely;
- unexpected or altered draft state denies automatic cleanup;
- source and artifacts are revalidated before and after publication;
- final published bytes and remote tag match the accepted source;
- private evidence is complete and censored; and
- no tag, branch, consuming project, or other repository is modified.
