# ISRAS 0.1.7 Release Notes

**Release date:** 2026-07-22

**Assurance status:** CORRECTIVE RELEASE CANDIDATE — SELF-VALIDATION REQUIRED

**Release tag:** `isras-v0.1.7`

## Purpose

ISRAS 0.1.7 preserves the complete Workstream A baseline and the 0.1.6
minimum-Go artifact correction. It corrects the reusable hosted adapter so
validator-owned `govulncheck` tooling no longer creates Git-visible files inside
a consuming repository before project-command execution.

## Corrected hosted tool boundary

The reusable hosted workflow now installs exact governed `govulncheck` under
runner-owned temporary storage. The release validator receives its absolute path
through a validator-owned environment boundary.

That path is accepted only when it is:

- absolute and normalized;
- outside the target repository;
- a regular executable file; and
- not a symbolic link.

Exact scanner module and version verification remains mandatory. Clean-tree
enforcement is not weakened, and consuming projects are not required to ignore
validator-owned tooling directories.

## Minimum Go boundary

The repository `go.mod` `go` directive remains a minimum:

- the exact minimum passes;
- later compatible patch and minor releases pass;
- valid custom-suffix toolchains pass;
- invalid identities fail; and
- selected toolchains below the minimum fail.

The actual selected Go toolchain remains bound into provenance.

## 0.1.6 disposition

`isras-v0.1.6` remains an immutable signed and published release. Its reusable
hosted adapter contains the target-owned tool-installation defect corrected by
0.1.7. The 0.1.6 tag and assets are not moved, replaced, or republished, and
0.1.6 is not adoption authority.

## Release completion

Release completion requires the exact signed candidate, hosted validation,
governed merge to `dev`, clean-clone release validation, signed annotated tag,
deterministic six-asset production and reproduction, separately reviewed tag
push, publication preflight, draft-first controlled publication, remote-byte
verification, and final release verification.

## Adoption readiness

A consuming repository may pin 0.1.7 only after the signed tag and verified
six-asset GitHub Release exist and that repository passes its own explicit
adoption or upgrade validation.

## Limitations

- The release remains self-authored and self-validated.
- No independent audit, certification, regulatory approval, or universal
  production-fitness claim is made.
- The validator release artifact remains linux/amd64.
