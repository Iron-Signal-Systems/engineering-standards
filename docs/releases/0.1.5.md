# ISRAS 0.1.5 Release Notes

**Release date:** 2026-07-21

**Assurance status:** RELEASE CANDIDATE — SELF-VALIDATION REQUIRED

**Release tag:** `isras-v0.1.5`

## Purpose

ISRAS 0.1.5 completes Workstream A of the repository-assurance baseline. The
release strengthens project-command execution, Go toolchain and multi-module
evidence, exact vulnerability validation, governed vulnerability exceptions,
repository identity, and fail-closed documentation-impact enforcement.

ISRAS defines and enforces the standard. Consuming repositories apply the
published release through separate, explicitly reviewed adoption or upgrade
workstreams.

## Go project-command boundary

This release candidate:

- preserves the active caller-selected Go executable while retaining a bounded
  child-process environment;
- enforces each governed `go.mod` `go` directive as a minimum version;
- accepts later compatible versions and valid custom-suffix toolchains;
- discovers every repository-owned Go module while excluding generated `.local`
  evidence;
- requires the selected Go implementation to satisfy every governed module; and
- records selected-Go, minimum-comparison, and per-module evidence through
  project-command execution schema version 2.

## Vulnerability validation

The Go-profile `known_vulnerabilities` command now:

- is fixed to exact `govulncheck ./...` semantics;
- verifies the governed scanner command path, module, version, build Go, and
  digest before execution;
- executes every governed module with `GOTOOLCHAIN=local` and `GOENV=off`;
- parses the scanner's streaming JSON protocol with bounded, fail-closed rules;
- retains typed per-module evidence; and
- fails on reachable or unknown findings unless an exact governed exception is
  valid and used.

Exceptions require exact finding scope, independent approval, expiration,
compensating controls, remediation, and deterministic used, unused, unexcepted,
or mismatched evidence.

## Governance and hosted enforcement

This release candidate also:

- defines ISRAS as the governing engineering authority for Iron Signal Systems
  repositories while distinguishing public visibility from product scope;
- affirms a language-neutral core with additive implementation profiles;
- enforces a versioned documentation-impact policy through exact Git
  merge-base comparison and retained JSON/text evidence;
- requires implementation, schema, workflow, release, and adoption changes to
  carry synchronized governance documentation; and
- validates native Ubuntu plus bind-mounted Arch Linux, Fedora 43, and Fedora 44
  environments, including exact-root Git ownership trust.

## Release completion

Release completion requires:

1. the exact signed 0.1.5 source commit merged to `dev`;
2. clean-clone release validation of the exact pushed commit;
3. a signed annotated `isras-v0.1.5` tag;
4. deterministic production and review of all six assets;
5. read-only publication preflight;
6. controlled draft-first publication with no-clobber uploads;
7. complete remote metadata and downloaded-byte verification; and
8. final post-publication re-verification.

A stable `VERSION`, branch head, or tag alone is not publication authority.

## Adoption readiness

Publication and consumer adoption are separate boundaries. A consuming
repository may pin 0.1.5 only after the signed tag and verified six-asset GitHub
Release exist and that repository passes its own required adoption or upgrade
validation.

## Limitations

- The release remains self-authored and self-validated.
- No independent audit, certification, regulatory approval, or universal
  production-fitness claim is made.
- The validator release artifact remains linux/amd64.
- Later defects require a later immutable release identity.
