# ISRAS 0.1.3 Release Notes

**Release date:** 2026-07-19

**Assurance status:** SELF-VALIDATED

**Release tag:** `isras-v0.1.3`

## Purpose

ISRAS 0.1.3 corrects defects discovered during the first real
consuming-project hosted-validation attempt against ISRAS 0.1.2. The 0.1.2
release remains immutable; these corrections are delivered only through a new
release identity.

The release remains a solo-developer, self-validated assurance baseline. It
does not claim independent audit, certification, regulatory approval, or
universal production fitness.

## Included corrections

This release includes:

- release-bound SSH allowed-signers trust sourced from the exact pinned
  Engineering Standards commit;
- committed trust inventory, complete SHA-256 binding, and explicit authority
  metadata;
- a private runtime trust copy configured before repository validation;
- exact committer-email, signer-principal, and signing-key fingerprint binding;
- fail-closed rejection of missing, altered, wrong-key, and wrong-principal
  trust;
- regression coverage for correct and hostile trust cases;
- retention of both `.local/isras` and `.local/validation` hosted evidence;
- survival of repository-signature failure logs when later workflow steps are
  skipped;
- GitHub Release asset uploads through the required `uploads.github.com` host;
- regression coverage requiring all six asset uploads to use the correct upload
  transport; and
- synchronized release, adoption, trust, evidence, and publication
  documentation.

## Exact release assets

The release contains exactly:

```text
isras-validator-linux-amd64
isras-project-framework.tar.gz
isras-contracts.tar.gz
provenance.json
SHA256SUMS
SHA512SUMS
```

No additional release asset is accepted. Every non-manifest asset is covered by
both manifests, and the manifest files are separately bound by the release
evidence and project pin.

## Release identity and completion

Release completion requires all of the following to identify the same exact
source commit:

1. stable `VERSION` value `0.1.3`;
2. verified signed release-source commit;
3. successful pull-request and merged-`dev` validation;
4. successful commit-mode and clean-clone release-mode validation;
5. signed annotated local tag `isras-v0.1.3`;
6. separately reviewed push of that exact tag object;
7. deterministic production and local verification of the six assets;
8. controlled draft creation and exact asset upload;
9. remote metadata and downloaded-byte verification before publication;
10. publication of the verified draft; and
11. final remote metadata and downloaded-byte verification after publication.

A stable source declaration, branch, tag name, workflow run, or GitHub Release
record is not sufficient by itself.

## Post-publication adoption readiness

Publication and project-adoption readiness are separate boundaries. After the
release is published, a neutral consuming-project fixture with an SSH-signed
commit must pin the exact 0.1.3 release and pass the exact reusable hosted
workflow on a clean GitHub-hosted runner.

That fixture must retain both evidence trees and prove that repository
signature verification, secret protection, project-pin verification, and every
declared project command pass through the exact published validator. A product
repository is not required to serve as the standard's test fixture.

## Project adoption status

ISRAS 0.1.3 may authorize a consuming-project adoption only after the exact
signed tag and six verified assets are published and the project's required
hosted validation passes on its exact adopted commit.

Local initialization, release verification, generated adoption files, or a
signed project commit do not constitute formal adoption when required hosted
validation fails.

ISRAS 0.1.2 remains published and immutable. Its failed hosted-validation
adoption candidate must not be reclassified as accepted merely because 0.1.3
later corrects the standard.

## Known limitations

- The release is self-authored and self-validated.
- The Go profile remains the first supported reference profile.
- Initialization supports clean repositories and does not migrate partial or
  conflicting adoption state.
- Project-upgrade application remains outside this release boundary.
- Trust authority is only as current as the signed release containing the
  pinned trust inventory; signer changes require a later reviewed release.
- Container-userland evidence does not establish complete native kernel,
  systemd, SELinux, firewall, hardware, deployment, or recovery compatibility.
- Release verification establishes exact identity and bytes, not suitability
  for every consuming project or operating environment.

## Recovery and immutability

Published tags, assets, manifests, provenance, framework contents, contracts,
and trust material are immutable. A defect in 0.1.3 must be corrected through a
later release rather than by replacing bytes under the existing identity.

Recovery consists of checking out the signed release tag or exact source
commit, rebuilding or reacquiring the declared assets, and verifying their
complete identity and digests. Private validation, build, command, adoption,
and publication evidence below `.local/` remains outside the signed source tree
and must be retained according to operational policy.
